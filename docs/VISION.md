## User as Shareholder Model

You are not a developer. You are not a manager. You are the **board of directors**.

### Interaction Levels

| Product Phase | Your Involvement | What You Do |
|---------------|------------------|-------------|
| **New Launch** | Deep | Set vision, approve architecture, guide direction |
| **Growing** | Regular | Review metrics, approve major features, course correct |
| **Mature** | Light | Weekly report review, approve budgets |
| **Declining** | Decision | Decide: pivot, invest more, or sunset |

### Communication Model

```
ACIA → You:
├── Daily: Critical alerts only (if any)
├── Weekly: Portfolio report
│   ├── Revenue per product
│   ├── Costs per product
│   ├── Key metrics changes
│   ├── Issues requiring attention
│   └── Opportunities identified
├── On-demand: Questions requiring decision
└── Async: Notifications you configure

You → ACIA:
├── Strategic direction: "Focus on B2B this quarter"
├── Product decisions: "Kill Product X, it's not working"
├── Approvals: "Yes, proceed with that feature"
└── Ideas: "I think there's opportunity in Y market"
```

### Confidence Factors

| Factor | High Confidence | Low Confidence |
|--------|-----------------|----------------|
| Similar changes before | Many successful | None/few |
| Test coverage | High | Low |
| Scope of change | Small, isolated | Large, cross-cutting |
| Pattern match | Known pattern | Novel approach |
| Rollback ease | Easy | Difficult |

Prevents "AI bloat" - unnecessary growth, complexity, or drift:


## Persona-Based QA

Traditional QA: Same agent writes code and tests → shared blind spots.

ACIA QA: Separate agents with NO shared context simulate real users.

### The Isolation Principle

```
┌─────────────────────────────────────────────────────────────────┐
│                    CONTEXT ISOLATION                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   DEV AGENT CONTEXT:              QA PERSONA CONTEXT:           │
│   ─────────────────               ──────────────────            │
│   ✓ Full implementation           ✓ Original requirement        │
│   ✓ Design decisions              ✓ The running app             │
│   ✓ Why choices were made         ✓ Persona expectations        │
│   ✓ Known limitations             ✗ NO implementation details   │
│   ✓ Technical constraints         ✗ NO dev reasoning            │
│                                   ✗ NO "why" explanations       │
│                                                                 │
│   Dev knows HOW it works          QA knows WHAT it should do    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Persona Types

| Persona | Behavior | Catches |
|---------|----------|---------|
| **Beginner** | Tries obvious things, gets confused easily | UX issues, missing guidance |
| **Expert** | Tries edge cases deliberately | Logic bugs, missing features |
| **Adversarial** | Tries to break things | Security issues, crashes |
| **Accessibility** | Uses keyboard, screen reader | A11y violations |
| **Impatient** | Expects instant response | Performance issues |
| **Mobile** | Uses on small screen | Responsive design issues |

### Oracle Testing

For domains with known-correct implementations, compare against them:

```
Building: Chess game
Oracle: Stockfish or chess.js

For every move:
  yourEngine.makeMove(move)
  oracle.makeMove(move)
  assert(yourEngine.position === oracle.position)

If they disagree → your implementation is wrong
```

This catches bugs that unit tests miss because the oracle knows the REAL rules.
